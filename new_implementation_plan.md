# TRPG 通用 Agent 新实施计划（GM 直控版）

## 1. 文档目标

这份新实施计划只保留旧实现计划中已经落地的基础能力，并在此基础上重新定义后续方向。

新的核心原则如下：

1. GM 直接接管战斗、叙事、场景推进和裁定，不再引入程序化的战斗状态机、场景状态机或独立 Orchestrator。
2. 系统唯一必须强程序化的部分是对话顺序、发言权限、notebook 访问边界，以及共享对话历史的维护与压缩。
3. 运行时架构应尽量保持 TRPG 通用性，不把 D&D 5e 特有的战斗回合、资源结算或状态推进硬编码进框架。
4. notebook 是核心外部状态载体，但 notebook 的维护主体是 GM 和各玩家 agent，而不是预设的程序化状态机。

## 2. 当前已实现的基础能力

本节只保留当前仓库中已经实现、可复用的部分。

### 2.1 文档预处理

以下能力已经实现，可作为后续 GM runtime 的资料输入底座：

1. `src/documents/extract.py` 已可将 PDF 资料转换为后续可消费的 Markdown。
2. 预处理流程已经转为 Markdown-first，而不是依赖旧的 JSONL-only 中间格式。
3. 输出 Markdown 保留统一页码标记，可供后续阅读工具和 RAG 使用。
4. `src/rag/test.py` 已能把 `documents/城主指南2024/` 这类离线网页目录转成单一 Markdown 文件。

### 2.2 Markdown-first RAG

以下检索基础设施已经实现：

1. `src/rag/rag.py` 已实现 Markdown-first RAG 主入口。
2. 当前支持索引构建、索引加载、通用查询、规则查询、世界观查询。
3. 切块策略已经支持按 Markdown 标题层级和页码标记进行分层切块。
4. 当前 embedding 已支持 OpenAI、HuggingFace，以及 DashScope OpenAI-compatible 的 Qwen embedding。
5. 检索结果保留 `doc_id`、`page_num`、`sections` 等元数据，便于 GM 回查原文。

### 2.3 资料阅读与检索工具

以下工具已经在 `src/tools/tools.py` 中实现：

1. `read_document_page(doc_id, page)`
2. `read_document_section(doc_id, section_title)`
3. `search_document(doc_id, query, top_k)`
4. `lookup_index(doc_id, keyword)`
5. `query_rules_tool(query, top_k)`
6. `query_lore_tool(query, top_k)`
7. `roll_dice(expression)`

这些工具已经足够支持以下 GM 工作方式：

1. 先搜索定位候选段落。
2. 再按页或章节回看原文。
3. 必要时再使用 RAG 获取更广义的相关片段。

### 2.4 基础 notebook 工具

以下 notebook 相关能力已经存在，但仍处于基础版：

1. `read_notebook(section, keys)`
2. `update_notebook(section, patch_json)`
3. `append_log(event)`
4. `summarize_log(window)`
5. `advance_turn()`

当前状态说明：

1. notebook 已经可以作为外部文件被读写，而不是只放在 prompt 中。
2. 目前 notebook 工具仍较基础，尚未支持搜索、章节跳转、访问隔离、多 notebook 角色分工等高级能力。
3. `advance_turn()` 已经存在，但后续只应保留为对话顺序推进工具，不应扩展为程序化战斗控制器。

### 2.5 Rule Retreival Agent

`src/agents/rule_retrieval.py` 已经实现为独立可用的规则专职 agent。

当前已经实现的能力包括：

1. `compile_rules_summary()`：开局规则整理。
2. `answer_rule_query()`：运行期规则调查。
3. 两阶段摘要流程：先由小模型生成 compressed rule notes，再由主模型整合成 GM 可读摘要。
4. 运行期规则调查可通过工具完成目录定位、关键词搜索、按页阅读和按章节阅读。
5. bootstrap / search prompt 已外置到 `prompts/` 目录。
6. 规则整理和规则调查都带有审计日志。

1. `prompts/rule_retreival_bootstrap.prompt`

当前测试重点覆盖：

## 3. 新的运行时架构原则

### 3.1 GM 直接控制

后续系统不再采用“独立 Orchestrator + GM”的结构，而改为由 GM 直接总控。

GM 的职责包括：

1. 维护整体游戏主循环。
2. 决定当前是谁发言。
3. 决定当前是否进入战斗、社交、探索或自由叙事。
4. 决定是否调用规则查询、骰子、notebook 等工具。
5. 扮演世界、NPC、敌人和环境反馈。
6. 自行处理战斗叙事、战斗节奏和裁定，而不是依赖程序化战斗状态机。

这里的关键原则是：

1. 战斗状态、场景状态、回合推进不作为框架级强约束逻辑存在。
2. 这些内容由 GM 通过叙事、工具调用和 notebook 更新来维护。
3. 这样同一套系统才能更容易迁移到不同 TRPG，而不是绑定某一套战斗规则。

### 3.2 对话系统是唯一必须程序化的运行时层

在新的架构里，唯一需要强程序化维护的是对话系统。

必须保证：

1. 所有 agent 只有在轮到自己时才能发言。
2. 如果 GM 没有手动指定，则发言顺序按固定顺序推进。
3. 如果 GM 指定了临时顺序，则严格按该顺序推进。
4. 临时顺序结束后，自动轮到 GM。
5. 在每次轮转前，应在 prompt 中告诉 GM 接下来的行动顺序，以便其决定是否修改。

### 3.3 玩家配置

当前目标玩家配置为：

1. 1 个真人玩家。
2. 3 个 LLM 玩家。
3. 1 个 GM。

约束如下：

1. LLM 玩家只有在轮到其发言时才能发言。
2. 真人玩家也应处于同一顺序系统中，只是在其回合由外部真实输入提供内容。
3. GM 始终拥有调整顺序、打断、插入旁白和结束某一轮顺序的权力。

### 3.4 插话与指定下一个说话者

对话系统应额外支持两个受控能力：

1. 插话：玩家可以通过工具请求插话，但只有在系统显式授予本次插话时才能发言。
2. 指定下一个说话者：如果一个玩家正在询问另一个玩家，他可以通过工具指定下一个说话的玩家。

这一机制的目标不是让玩家自由抢答，而是让“定向追问”和“短打断”成为受控的对话动作。

## 4. notebook 新要求

### 4.1 notebook 是主要状态载体

新的架构中，战斗、场景、角色关系、角色成长、重大挫折、高光时刻等，都优先通过 notebook 维护。

这里不再追求一套程序化战斗状态对象，而是追求：

1. GM 可以方便查阅与更新。
2. 玩家 agent 可以维护自己的持续记忆。
3. 内容能长期积累，并能被搜索、跳转、压缩和引用。

### 4.2 每个玩家有多个 notebook

每个玩家不应只有单一本子，而应至少有多个预创建 notebook，例如：

1. `character_sheet.md`：角色卡。
2. `events.md`：高光时刻，角色弧光与长期目标，重大挫折与失败经验。
3. `private_notes.md`：该角色已知但未必要公开的个人记录。

这些 notebook 应提前创建好，由 agent 自行维护，而不是运行时临时发明结构。

### 4.3 notebook 权限边界

权限模型应明确如下：

1. 每个玩家只能读取和修改自己的 notebook。
2. 玩家不能直接读取或修改其他玩家的个人 notebook。
3. GM 可以读取和修改任何人的 notebook。
4. 例如，当玩家损失血量时，GM 可以直接修改该玩家角色卡上的 HP。

### 4.4 notebook 高级功能

后续 notebook 工具必须扩展出以下能力：

1. 搜索。
2. 按章节跳转。
3. 定点更新。
4. 追加记录。
5. 摘要与压缩。
6. 限制访问范围的读写接口。

## 5. 对话历史与知识边界

### 5.1 共享对话记录

所有参与者共享一份对话记录，该记录会持续追加，并定时压缩。

共享记录的目标是：

1. 保留最近游戏上下文。
2. 降低上下文窗口成本。
3. 让 agent 可以基于共同经历继续对话。

### 5.2 定时压缩

系统应支持对共享对话记录进行周期性压缩。

压缩结果应满足：

1. 便于后续快速恢复上下文。
2. 不需要每次把完整长记录重新塞回 prompt。
3. 保留行动顺序、关键承诺、重要冲突、玩家关系变化等长期有效信息。

### 5.3 信息边界与“装傻”原则

虽然共享对话记录是公共的，但 agent 不应该因此自动获得一切可行动知识。

因此需要两层约束：

1. 工具层做硬边界：个人 notebook 访问受权限限制。
2. prompt 层做软边界：要求 agent 对自己不应知道或不应利用的信息“装傻”，避免越权行动。

这意味着：

1. 某些内容即使出现在压缩摘要或共享历史中，玩家 agent 也应被提示不要用其做出超出角色认知范围的判断。
2. GM 是例外，GM 可以基于全局信息进行管理和裁定。

## 6. Rule Retreival Agent 在新架构中的位置

Rule Retreival Agent 继续存在，但角色要被重新限定为“规则专职辅助模块”。

它负责：

1. 开局前整理系统规则，生成 GM 可读规则摘要。
2. 运行期回答规则问题，为 GM 提供引用和裁定参考。

它不负责：

1. 主持叙事。
2. 维护对话顺序。
3. 接管战斗流程。
4. 管理玩家发言权。

也就是说，在新架构中，Rule Retreival Agent 是 GM 的工具，而不是 GM 的上层控制器。

## 7. 后续开发顺序

在新的架构下，后续开发应按下面顺序推进：

1. 保留当前已实现的文档预处理、RAG、阅读工具、Rule Retreival Agent 和基础 notebook 工具。
2. 重构 runtime 目标，不再引入程序化 combat / scene engine。
3. 先实现对话顺序系统，包括默认顺序、GM 临时改序、插话和指定下一个说话者。
4. 再扩展 notebook 工具，包括多 notebook、访问隔离、搜索、章节跳转、定点更新和摘要。
5. 再补共享对话记录与定时压缩能力。
6. 然后把 1 个真人玩家 + 3 个 LLM 玩家正式接入。
7. 最后把 Rule Retreival Agent 作为规则辅助模块整合进 GM runtime。

## 8. 最小可行目标

新的最小可行目标不是“做一个程序化战斗系统”，而是：

1. GM 可以直接主持不同 TRPG 的叙事、探索和战斗。
2. 系统可以约束谁当前能发言。
3. 玩家只能在轮到自己时发言，除非被允许插话或被指定为下一个说话者。
4. GM 总能看到并调整接下来的发言顺序。
5. 每个玩家拥有多个独立 notebook，并只能维护自己的 notebook。
6. GM 可以修改任何人的 notebook。
7. 共享对话记录能够持续累积并定期压缩。
8. Rule Retreival Agent 能继续提供规则摘要和规则问答支持。

达到这一目标后，系统就已经具备跨 TRPG 可迁移的 GM 主导游玩能力。
